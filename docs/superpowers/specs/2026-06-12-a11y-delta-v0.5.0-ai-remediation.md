# a11y-delta v0.5.0 — AI Remediation Design

**Date:** 2026-06-12
**Status:** Approved
**Milestone:** v0.5.0

---

## Problem

v0.1.0–v0.4.0 tell you *what* is broken and *where*. They do not tell you *how to fix it* — that gap requires a developer to cross-reference WCAG docs, understand the element's computed styles, and write the patch. AI can collapse that lookup into a single code suggestion.

---

## Goals

- Per-violation AI-generated code fix — ≤3 lines, no prose
- Provider-agnostic: OpenAI, Anthropic, Google, Mistral, Ollama, any OpenAI-compatible endpoint
- Fix appears in **both** terminal output and HTML report simultaneously
- Provider/model configured in `.a11y-delta.yml` (team defaults) with full CLI override
- API keys from env vars — never logged, never embedded in output files
- `--ai` is opt-in; all prior behaviour unchanged without the flag

---

## What Does Not Change

Everything in v0.1.0–v0.4.0 is unchanged. `--ai` is purely additive — it enriches all active outputs that support it.

## Supported Providers

| Provider | `--ai-provider` value | Key env var | Notes |
|---|---|---|---|

## `.a11y-delta.yml` — `ai:` section

Extends the existing config file from v0.2.0:

```yaml
# .a11y-delta.yml
base: https://staging.example.com
pages:
  - /
  - /products/

# New in v0.5.0:
ai:
  provider:   openai           # openai | anthropic | google | mistral | ollama | openai-compatible
  model:      gpt-4o-mini      # optional — falls back to provider default
  base-url:   null             # required for ollama and openai-compatible
  max-fixes:  10               # max AI calls per run (default: 10; prevents runaway cost)
  priority:   critical         # critical | serious | all (default: critical)
                               # controls which violations receive AI fixes
  timeout-ms: 15000            # per-request timeout (default: 15000)
```

**CLI always overrides config** (same precedence as all other flags):

```bash
# Override provider + model for a one-off run:
npx a11y-delta --ai --ai-provider anthropic --ai-model claude-sonnet-4-6 ...

# Use config defaults (no flags needed):
npx a11y-delta --ai ...
```

**`max-fixes` cost guard:** if more violations exist than `max-fixes`, the highest-impact violations are prioritised (critical first, then serious). Remaining violations are shown without AI fixes — no error, just a footer note: `(N violations without AI fix — increase ai.max-fixes to cover all)`.

## New CLI Flags

```
AI remediation:
  --ai                        Enable AI fix suggestions (requires provider config)
  --ai-provider <name>        Provider: openai | anthropic | google | mistral | ollama | openai-compatible
  --ai-model    <model>       Model name (uses provider default if omitted)
  --ai-base-url <url>         Base URL for ollama or openai-compatible providers
  --ai-max-fixes <n>          Override ai.max-fixes from config
  --ai-priority <level>       Override ai.priority: critical | serious | all
```

API keys are **never passed as CLI flags** — always read from standard env vars per provider. This ensures keys are not visible in `ps aux`, shell history, or CI logs.

## Prompt Structure

The same prompt is used for all providers. It is designed for concise, usable output.

**System prompt (invariant across all violations):**
```
You are an accessibility engineer reviewing HTML. When given a failing axe-core
accessibility violation, return ONLY the minimal code change needed to fix it.

Rules:
- Return code only. No explanation, no prose, no markdown fences.
- Maximum 3 lines. Prefer 1 line when possible.
- Use the element's actual computed values (from the "axe data" field) when available.
- If you cannot determine a safe fix, return the string: UNKNOWN
```

**User message (per violation):**
```
Accessibility violation on {candidateUrl}

Rule:    {ruleId}  ({wcagCriterion}, {impact})
Element: {target.join(', ')}
HTML:    {node.html}
Issue:   {node.failureSummary}
axe data: {JSON.stringify(node.any[0]?.data ?? {})}
Page context (parent chain, 3 levels):
{parentHtml}
```

**Response handling:**
- Strip leading/trailing whitespace
- If response is `UNKNOWN` or empty → `fix: null` (no fix shown)
- If response exceeds 10 lines → truncate to first 3 lines + append `/* … */`
- Response is stored as-is in the violation's `fix.aiSuggestion` field — never executed

**Token estimate per violation:** ~400–800 input tokens. At `max-fixes: 10`, worst case ~8,000 input tokens per run — negligible cost on any provider.

## AI Provider Abstraction Layer — `src/ai.js`

All providers are normalised behind a single interface:

```javascript
// src/ai.js
export async function getAiFix(violation, config) {
  const { provider, model, baseUrl, timeoutMs } = config;
  const prompt = buildPrompt(violation);

  switch (provider) {
    case 'openai':
    case 'openai-compatible':
      return callOpenAiCompatible(prompt, model, baseUrl ?? 'https://api.openai.com/v1', timeoutMs);
    case 'anthropic':
      return callAnthropic(prompt, model, timeoutMs);
    case 'google':
      return callGoogle(prompt, model, timeoutMs);
    case 'mistral':
      return callOpenAiCompatible(prompt, model, 'https://api.mistral.ai/v1', timeoutMs);
    case 'ollama':
      return callOpenAiCompatible(prompt, model, baseUrl ?? 'http://localhost:11434/v1', timeoutMs);
    default:
      throw new Error(`Unknown AI provider: "${provider}". Use: openai, anthropic, google, mistral, ollama, openai-compatible`);
  }
}
```

All providers return a `{ suggestion: string | null, tokensUsed: number }` shape. HTTP is handled via `node:https` — no fetch polyfill, no SDK dependencies.

**No npm dependencies added for AI.** All providers are called via their HTTP API directly. This keeps the package lightweight and avoids SDK version lock-in.

---

## Concurrency

AI calls run concurrently, capped at `min(max-fixes, 5)` simultaneous requests, using the same `src/pool.js` from v0.2.0.

Sequential processing (not parallel) within a page — this matches the violation display order and makes terminal output predictable.

## Privacy & Security

- API keys read exclusively from env vars — `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`, `MISTRAL_API_KEY`, `AI_API_KEY`
- Keys are never passed as CLI flags (not in the flag definitions)
- Keys are never written to report files, JSON output, or log output
- The element's `node.html` (outerHTML) and `parentHtml` are sent to the LLM — teams should be aware this may include content from the page. The `--ai-privacy-mode` flag (future) would strip all page content and send only the rule ID + element type.
- Ollama calls stay entirely local — no data leaves the machine

## Error Handling

| Scenario | Behaviour |
|---|---|
| `--ai` set but no provider key in env | stderr: `"Error: OPENAI_API_KEY is not set. Set it or use --ai-provider to choose a different provider."`, exit 2 |
| `--ai` set but no `ai.provider` in yml and no `--ai-provider` flag | stderr: `"Error: --ai requires a provider. Set ai.provider in .a11y-delta.yml or pass --ai-provider."`, exit 2 |
| Provider HTTP call returns 401 | `fix.aiSuggestion = null`, stderr warning: `"AI fix skipped for color-contrast: 401 Unauthorized"`, run continues |
| Provider timeout | `fix.aiSuggestion = null`, no error (silently skipped) |
| Ollama not running | Connection refused → `fix.aiSuggestion = null`, stderr warning once: `"Ollama not reachable at http://localhost:11434 — AI fixes skipped"` |
| `max-fixes` cap reached | Remaining violations get `fix.aiSuggestion = null`, footer note: `(5 violations without AI fix — increase ai.max-fixes to cover all)` |
| LLM returns empty string | Treated same as `UNKNOWN` → `fix.aiSuggestion = null` |

**Key principle:** AI failures are never fatal. A network error or bad response downgrades to no AI fix — the accessibility diff still runs, violations still reported, exit codes still correct.

## Self-review fixes applied

**Placeholder scan:** None.

**Consistency:** `fix.aiSuggestion` field is present on every violation entry in the internal shape after `multi.js` runs (set to `null` when not generated). All render functions check for `null` before showing the AI box — no conditional rendering bugs possible from undefined fields.

**Scope:** Single implementation plan covers `ai.js` (provider layer) + `config.js` extension + `multi.js` AI enrichment loop + render updates across 4 output formats. Appropriately scoped for one milestone.

**Privacy note documented:** The prompt sends `node.html` to external APIs. This is called out explicitly in the spec so it is a conscious decision, not an oversight.
