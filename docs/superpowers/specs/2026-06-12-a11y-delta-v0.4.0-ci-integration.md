# a11y-delta v0.4.0 — CI Integration Depth Design

**Date:** 2026-06-12
**Status:** Approved
**Milestone:** v0.4.0

---

## Problem

v0.3.0 produces a standalone `report.html` and a GitHub PR comment. Teams also need violations surfaced in Slack/Teams channels, inline on the Files Changed tab, and in the Actions Step Summary — without writing custom CI scripts.

---

## Goals

- Three-layer GitHub output: PR comment + Step Summary + best-effort file annotations
- Slack notification: rich Block Kit card with stat counters + PR/report links
- Teams notification: Adaptive Card with same data
- README badge: dynamic SVG updated on each CI run
- All opt-in via CLI flags — zero config required for teams that don't want these

---

## What Does Not Change

All existing outputs (`--format table/github-comment/json`, `--report`, `--save`, `--save-dir`) unchanged. New flags are purely additive.

---

## New CLI Flags

```
GitHub CI integration:
  --gh-token       <token>    GitHub token (reads from GITHUB_TOKEN env if not set)
  --gh-annotations            Post violations as GitHub Check annotations on Files Changed
                              (best-effort selector→file mapping; silently skips when not found)
  --gh-step-summary           Write full violation table to $GITHUB_STEP_SUMMARY
  --pr-comment                Post summary+collapsible PR comment (uses --format github-comment output)
                              Reads PR number from GITHUB_REF / PR_NUMBER env automatically

Notifications:
  --slack-webhook  <url>      Post Block Kit card to Slack webhook
  --teams-webhook  <url>      Post Adaptive Card to Teams webhook

Badge:
  --badge          <path>     Write dynamic SVG badge to file (default: .a11y-badge.svg)
                              Red = critical violations, orange = serious-only, green = clean

Convenience:
  --ci                        Shorthand for: --gh-step-summary --pr-comment
                              (the two safest flags — no token scopes beyond write:checks)
```

### Example GitHub Actions step

```yaml
- name: Accessibility delta check
  run: |
    npx a11y-delta \
      --sitemap https://staging.example.com/sitemap.xml \
      --candidate-base https://preview-${{ github.event.pull_request.number }}.example.com \
      --ci \
      --gh-annotations \
      --slack-webhook ${{ secrets.SLACK_A11Y_WEBHOOK }} \
      --badge .a11y-badge.svg \
      --report a11y-report.html
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

- name: Upload report artifact
  uses: actions/upload-artifact@v4
  if: always()
  with:
    name: a11y-report
    path: a11y-report.html
```

## Layer 1: PR Comment (`--pr-comment`)

Reuses `renderGithubCommentMulti` from v0.2.0. Posts (or updates) a PR comment via GitHub REST API.

**Logic:**
1. Look for an existing comment by `a11y-delta` bot on the PR (search comment bodies for `<!-- a11y-delta -->` marker)
2. If found: update it (`PATCH /repos/:owner/:repo/issues/comments/:id`)
3. If not found: create it (`POST /repos/:owner/:repo/issues/:number/comments`)

**PR number resolution order:**
1. `--gh-pr-number` flag
2. `PR_NUMBER` env var
3. `GITHUB_REF` → parse `refs/pull/N/merge`
4. `GITHUB_EVENT_PATH` → read `pull_request.number` from event JSON

If PR number cannot be resolved: skip PR comment with stderr warning, do not exit 2.

**Comment body:**

```markdown
<!-- a11y-delta -->
## ♿ Accessibility Delta — 3 new violations across 2 pages

| Page | New | Worst |
|---|---|---|
| /products/ | 2 | critical |
| /checkout/ | 1 | critical |
| / | 0 | — |

<details>
<summary>📄 /products/ — 2 new violations</summary>
...
</details>

**3 pages audited · concurrency: 3 · [Full report](./a11y-report.html)**
```

## Layer 2: Step Summary (`--gh-step-summary`)

Writes to `$GITHUB_STEP_SUMMARY` (the GitHub Actions job summary).

No API call needed — just `fs.appendFileSync(process.env.GITHUB_STEP_SUMMARY, content)`.

**Content:** Full multi-page violation table (same as `--format github-comment`) plus a link to the uploaded report artifact if `--report` was also passed.

If `GITHUB_STEP_SUMMARY` env var is not set (not running in GitHub Actions): skip silently, no error.

## Layer 3: File Annotations (`--gh-annotations`)

Posts violations as GitHub Check annotations — they appear as inline comments on the Files Changed tab pointing at the source file that contains the offending element.

### Selector → file mapping

```javascript
// src/annotations.js
async function mapSelectorToFile(selector, repoRoot) {
  // Extract meaningful tokens from CSS selector
  // e.g. "button.add-to-cart" → ["add-to-cart", "add_to_cart", "addToCart"]
  const tokens = extractTokens(selector);

  for (const token of tokens) {
    // Grep for token in source files (*.jsx, *.tsx, *.vue, *.html, *.svelte, *.erb)
    const matches = await grepFiles(repoRoot, token, [
      '**/*.jsx','**/*.tsx','**/*.vue','**/*.html',
      '**/*.svelte','**/*.erb','**/*.hbs','**/*.njk',
    ]);
    if (matches.length === 1) return matches[0]; // { file, line }
    if (matches.length > 1) return matches[0];   // take first match, warn if ambiguous
  }
  return null; // not found — skip annotation for this violation
}
```

**Token extraction rules:**
- `button.add-to-cart` → `["add-to-cart", "add_to_cart", "addToCart"]`
- `input[name="quantity"]` → `["quantity"]`
- `img.payment-icon` → `["payment-icon", "payment_icon", "paymentIcon"]`
- Generic selectors (single tag like `button`, `input`) → skip (too many false positives)

**When no file match is found:** violation is silently skipped for file annotation. A summary Check annotation is still posted at the repo root level listing all unmatched violations.

### GitHub Checks API usage

```javascript
// Create a Check Run and post annotations
const checkRun = await octokit.checks.create({
  owner, repo,
  name: 'a11y-delta',
  head_sha: process.env.GITHUB_SHA,
  status: 'completed',
  conclusion: newViolations.length > 0 ? 'failure' : 'success',
  output: {
    title: `${totalNew} new accessibility violation${totalNew !== 1 ? 's' : ''}`,
    summary: buildCheckSummary(multiResult),
    annotations: buildAnnotations(multiResult, fileMappings), // max 50 per request
  },
});
```

**Annotation shape:**

```javascript
{
  path:              'src/components/ProductCard.jsx',  // relative to repo root
  start_line:        42,
  end_line:          42,
  annotation_level:  violation.impact === 'critical' ? 'failure' : 'warning',
  title:             `a11y: ${violation.id} — ${violation.impact}`,
  message:           `${violation.description}\nElement: ${violation.target.join(', ')}\nWCAG: ${wcagCriterion}`,
  raw_details:       violation.failureSummary,
}
```

**Batching:** GitHub accepts max 50 annotations per Check Run update. If `totalNew > 50`, split into multiple `checks.update()` calls.

**Required token scope:** `checks:write` (included in the default `GITHUB_TOKEN` when the workflow has `permissions: checks: write`).

## Slack Integration (`--slack-webhook <url>`)

Posts a Block Kit message to the provided Incoming Webhook URL.

**Block Kit payload:**

```json
{
  "blocks": [
    {
      "type": "header",
      "text": { "type": "plain_text", "text": "♿ a11y-delta — 3 new accessibility violations" }
    },
    {
      "type": "section",
      "fields": [
        { "type": "mrkdwn", "text": "*Critical*\n:red_circle: 2" },
        { "type": "mrkdwn", "text": "*Serious*\n:large_orange_circle: 1" },
        { "type": "mrkdwn", "text": "*Clean pages*\n:white_check_mark: 1" },
        { "type": "mrkdwn", "text": "*Pages audited*\n:globe_with_meridians: 3" }
      ]
    },
    {
      "type": "section",
      "text": { "type": "mrkdwn", "text": "*Pages with new violations:* </products/> · </checkout/>" }
    },
    {
      "type": "actions",
      "elements": [
        { "type": "button", "style": "danger",   "text": { "type": "plain_text", "text": "View PR →" },     "url": "<PR_URL>" },
        { "type": "button", "style": "primary",  "text": { "type": "plain_text", "text": "Open Report" },   "url": "<REPORT_URL>" }
      ]
    }
  ]
}
```

**When `totalNew === 0`:** Posts a compact green success message instead of the full card.

**`--slack-webhook` URL is never logged** — passed directly to `node:https` POST, not printed to stdout/stderr.

## Badge (`--badge <path>`)

Generates a Shields.io-style SVG badge and writes it to `<path>` (default: `.a11y-badge.svg`).

**Colour logic:**

| State | Label text | Colour |
|---|---|---|
| `totalNew > 0` and `byCritical > 0` | `{N} critical` | `#e05252` (red) |
| `totalNew > 0` and `byCritical === 0` | `{N} serious` | `#f6993f` (orange) |
| `totalNew === 0` | `passing ✓` | `#4caf50` (green) |

**SVG template** (self-contained, no external resources):

```svg
<svg xmlns="http://www.w3.org/2000/svg" width="108" height="20">
  <rect width="28" height="20" fill="#555"/>
  <rect x="28" width="80" height="20" fill="${colour}"/>
  <text x="14" y="14" fill="#fff" font-family="monospace" font-size="11" font-weight="600" text-anchor="middle">a11y</text>
  <text x="68" y="14" fill="#fff" font-family="monospace" font-size="11" font-weight="600" text-anchor="middle">${label}</text>
</svg>
```

Width is calculated dynamically based on label text length.

**README embed:**

```markdown
[![a11y](.a11y-badge.svg)](https://github.com/your/repo/actions)
```

Commit `.a11y-badge.svg` to the repo. Update it by re-running `a11y-delta ... --badge .a11y-badge.svg` and committing the changed file in CI.

## Error Handling

| Scenario | Behaviour |
|---|---|
| `GITHUB_TOKEN` not set and `--gh-annotations` passed | stderr warning, annotation skipped, run continues |
| GitHub API call fails (rate limit, 403) | stderr warning with status code, annotation skipped, exit code unchanged |
| Slack/Teams webhook returns non-2xx | stderr warning `"Slack notification failed: HTTP 400"`, run continues |
| `--badge` path directory doesn't exist | `mkdir -p` before write (same pattern as `writeSnapshot`) |
| `$GITHUB_STEP_SUMMARY` not set | skip silently (not running in GitHub Actions) |
| PR number not resolvable | skip PR comment, stderr warning |
| Selector maps to 0 files | skip file annotation, include in fallback Check summary |
| Selector maps to 5+ files | skip file annotation (too ambiguous), include in fallback Check summary |

**Key principle:** All CI integrations are best-effort — a notification failure never changes the exit code or prevents other outputs from being written.

## Testing Approach

- `badge.test.js` — `generateBadge(result)` returns valid SVG string; correct colour for critical/serious-only/clean; label text matches counts; width scales with label length
- `annotations.test.js` — `extractTokens('button.add-to-cart')` returns correct variants; `mapSelectorToFile` with mock filesystem returns correct file+line; generic-tag selectors return null; ambiguous (5+ matches) returns null
- `notifications.test.js` — `buildSlackPayload(result)` returns valid Block Kit JSON shape; `buildTeamsPayload(result)` returns valid Adaptive Card JSON shape; zero-violations triggers success message variant
- `github.test.js` — `buildAnnotations(multiResult, mappings)` batches correctly at 50; annotation level is `failure` for critical, `warning` for others; `resolvePrNumber()` correctly parses each env var source
- Integration: mock HTTP server receives correct webhook payload when `--slack-webhook http://localhost:PORT` is passed
