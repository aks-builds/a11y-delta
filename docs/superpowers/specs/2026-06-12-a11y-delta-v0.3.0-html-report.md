# a11y-delta v0.3.0 — HTML Visual Report Design

**Date:** 2026-06-12
**Status:** Approved
**Milestone:** v0.3.0

---

## Problem

v0.1.0/v0.2.0 produce terminal table, GitHub comment markdown, and JSON. None of these help a developer visualise *where* on the page a violation appears or hand evidence to a designer/PM. A generated HTML report with element screenshots bridges that gap.

---

## Goals

- Single self-contained `report.html` — no CDN, no external assets, no server
- Three views in one file: **Screenshots**, **Table**, **Summary/print**
- Per-violation Playwright screenshots with the offending element highlighted
- Suggested code fix for each violation
- Works for single-URL (v0.1.0 mode) and multi-page (v0.2.0 mode)
- Sharable: open in any browser, attach to Jira/Linear, print to PDF

---

## What Does Not Change

- `src/diff.js`, `src/audit.js`, `src/snapshot.js` — unchanged
- All existing `--format table/github-comment/json` outputs — unchanged
- `src/render.js` gets one new export: `renderHtml(result, opts)`

## CLI Flag

```bash
# Generate HTML report (single URL)
a11y-delta \
  --baseline https://staging.example.com \
  --candidate https://preview.example.com \
  --report report.html

# Multi-page (v0.2.0)
a11y-delta \
  --sitemap https://staging.example.com/sitemap.xml \
  --candidate-base https://preview.example.com \
  --report report.html

# Combine with JSON (both outputs)
a11y-delta ... --format json --report report.html

# Open immediately after generation (macOS/Linux)
a11y-delta ... --report report.html --open
```

`--report <path>` triggers HTML generation. It is additive — `--format json` or `--format table` still work alongside it. `--open` calls `open`/`start`/`xdg-open` on the generated file after writing.

## Element Screenshots

Each violation gets a cropped Playwright screenshot of the offending element in its page context.

**Capture strategy:**

```javascript
// In audit.js — called per violation node after axe.run()
async function captureViolationScreenshot(page, targetSelector) {
  try {
    const el = page.locator(targetSelector).first();
    const box = await el.boundingBox();
    if (!box) return null;
    // Pad 16px around element for context
    const clip = {
      x:      Math.max(0, box.x - 16),
      y:      Math.max(0, box.y - 16),
      width:  Math.min(box.width + 32, page.viewportSize().width),
      height: Math.min(box.height + 32, page.viewportSize().height),
    };
    const buffer = await page.screenshot({ clip, type: 'webp', quality: 80 });
    return `data:image/webp;base64,${buffer.toString('base64')}`;
  } catch {
    return null; // screenshot failure never fails the audit
  }
}
```

Screenshots are embedded as `data:image/webp;base64,...` URIs in the HTML — no external files.

If screenshot capture fails (element not visible, selector invalid, etc.) the screenshot pane falls back to a styled placeholder showing the element's outerHTML in a code block.

**Highlight overlay:** A CSS `<div>` with `position:absolute; inset:0; border:1.5px dashed #ff4444` is positioned over the screenshot `<img>` (which is `position:relative`). This provides the red dashed highlight without canvas manipulation.

## Three Views

### View 1 — Screenshots (default tab)

Per-violation cards, per-page sections. Each card has:

**Card header (`var(--s2)` background):**
- Impact pill (Critical/Serious/Moderate/Minor) with colour token
- Rule name (monospace)
- Short description (truncated at 60 chars)
- WCAG criterion chip (right-aligned)

**Card body (two columns, `grid-template-columns: 192px 1fr`):**

Left — screenshot pane:
- Browser chrome mock (macOS traffic lights, URL bar)
- Cropped element screenshot (or HTML fallback)
- Caption: `elementSelector · /page-path/`

Right — detail pane:
- **Element** row: selector in `--link` code chip
- **Issue** row: plain-English description with `--crit`/`--ser` highlighted ratio
- **Suggested fix** box: `background:var(--s2); border-left:2px solid var(--ok)` — code snippet with `from` (line-through) and `to` (green)
- **Links** row: `axe docs`, `WCAG spec`, and context-specific links (e.g. "Contrast checker" for color-contrast)

### View 2 — Table tab

Flat filterable/sortable table. All violations across all pages in one view.

**Columns:** Impact · Rule · Element · Page · WCAG · (expand row icon)

**Interactions (pure CSS + HTML — no JS framework):**
- `<details>`/`<summary>` expand each row to show description + fix
- Sort by column: clicking a column header re-renders the table (JS required for sort — only interactive feature needing JS)
- Column headers show `▲`/`▼` sort indicators

**CSV export:** `<a href="data:text/csv;...">` link — generates CSV from table data at render time and embeds as a data URI. No JS needed for download.

### View 3 — Summary tab (print-ready)

Light background (`#ffffff`) — designed for printing or PDF export.

Sections:
1. **Header** — tool name + run metadata + date
2. **Stats row** — 4 number cards: Critical, Serious, Pages, Clean
3. **Per-page tables** — compact rule/element/fix table per failing page
4. **Footer** — "Generated by a11y-delta v0.3.0 · axe-core 4.10 · WCAG 2.1 AA"

`@media print` CSS: hides topbar, sidebar, tabs; shows only summary view content; uses white background, black text, 11pt font, standard page margins.

## Sidebar

Fixed 200px, `background:var(--s1)`, `border-right:1px solid var(--b1)`.

Sections (top to bottom):
1. **Search box** — filters visible violations by rule name or element selector (pure JS `querySelectorAll` + class toggle)
2. **Impact filter** — All / Critical / Serious / Moderate / Minor with counts

## Self-Contained HTML Architecture

The entire report is one `report.html` file. Generation strategy:

```javascript
// src/render.js — renderHtml(result, opts)
export function renderHtml(result, opts = {}) {
  const css    = buildCss();          // all styles as a single <style> block
  const js     = buildJs();           // ~80 lines: tab switching, filter, sort, scroll-to-page
  const body   = buildBody(result);   // per-page sections from PageResult[] or single-URL result
  return `<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width,initial-scale=1">
<title>a11y-delta report — ${new Date(result.timestamp).toISOString().slice(0, 10)}</title>
<style>${css}</style>
</head>
<body>
${body}
<script>${js}</script>
</body>
</html>`;
}
```

No bundler, no build step. The file is generated in memory and written with `fs.writeFile`. Typical file size: 200–800 KB (dominated by base64 screenshots). For large sites (50+ pages), screenshots are lazy-decoded by the browser — no performance issue.

## Suggested Fix Generation

Fixes are generated per rule using a lookup map in `src/fixes.js`:

```javascript
// src/fixes.js
export const FIX_HINTS = {
  'color-contrast': (node) => ({
    code: `color: ${node.any?.[0]?.data?.fgColor ?? '#cccccc'}  →  #1a1a1a\n/* or increase background lightness */`,
    links: ['axe docs', 'Deque University', 'Contrast checker'],
  }),
  'aria-label': () => ({
    code: `aria-label="Descriptive label"  /* add to element */\n/* or wrap in <label>...</label> */`,
    links: ['axe docs', 'WCAG 1.3.1'],
  }),
  'image-alt': () => ({
    code: `alt="Description of image"\n/* or alt="" if purely decorative */`,
    links: ['axe docs', 'WCAG 1.1.1', 'Alt text guide'],
  }),
  // ... ~30 most common axe rules covered
};

export function getFix(ruleId, node) {
  return FIX_HINTS[ruleId]?.(node) ?? {
    code: null,  // no fix hint — show axe failureSummary instead
    links: [`https://dequeuniversity.com/rules/axe/4.10/${ruleId}`],
  };
}
```

v0.3.0 ships with fix hints for the 30 most common axe rules by frequency. Unknown rules fall back to showing axe's own `failureSummary` text.

## Testing Approach

- `fixes.test.js` — `getFix(ruleId, node)` returns `{ code, links }` for all 30 covered rules; returns `{ code:null, links:[url] }` for unknown rule
- `render.test.js` — `renderHtml()` returns valid HTML string (parse with `node:html` or regex check for doctype/closing tags); contains all violation rule names; contains `data:image/webp` when screenshots provided; omits screenshot `<img>` when null (fallback path shown)
- `tests/fixtures/report-snapshot.html` — golden file: run `renderHtml(fixtureResult)` and compare output to stored snapshot (update with `UPDATE_SNAPSHOTS=1`)
- Integration: `a11y-delta --baseline baseline.json --candidate candidate-new.json --report /tmp/test-report.html` exits 1, file exists, file size > 1KB

Screenshots are not tested in unit tests (no real Playwright in CI). `captureViolationScreenshot` is tested via a mock that returns a fixed base64 string.

---

## Error Handling

- Screenshot capture throws → `null` returned → card shows HTML fallback (outerHTML in `<code>` block). Never fails the audit.
- `--report` path directory does not exist → `writeSnapshot`-style `mkdir -p` before write
- `--open` flag on unsupported platform (no `open`/`start`/`xdg-open`) → stderr warning, file still written
- `renderHtml` called with empty `pages[]` → renders a "No violations found" full-page success state (green checkmark, run metadata)
